## 2. Critical

### C1. NAS containers running outside Terraform (`speed-test`, `tg-fresh-marmoset`, `netbootxyz`)

- **What it is:** Three containers on `nas` running via Synology
  Container Manager projects, NOT via the Portainer agent → so
  Portainer doesn't see them and Terraform can't manage them.
  - `openspeedtest` (compose at `/volume2/docker/composer/speed-test/compose.yaml`)
  - `tg-fresh-marmoset` (twingate connector, compose at
    `/volume2/docker/composer/twingate-connector/compose.yaml`)
  - `netbootxyz` (compose at `/volume2/docker/netbootxyz/docker-compose.yaml`)
- **Why it's a gap:**
  1. **Plaintext secrets in repo & host:** the NAS twingate compose
     has `TWINGATE_ACCESS_TOKEN=eyJhbG...` and `TWINGATE_REFRESH_TOKEN=rk_VgUhO...`
     hardcoded in the YAML on disk at `/volume2/docker/composer/twingate-connector/compose.yaml`.
     This is the same anti-pattern called out for `feedback_env_file_credentials.md`.
     Token is also visible in `nas/docker/twingate-connector/compose.yaml` in the
     repo if it was ever committed (need to check git history; out of scope here).
  2. If the NAS is rebuilt or the SD image swapped, none of these
     come back — they have no IaC source.
  3. The `nas/docker/{netbootxyz,paperlessngx,portainer-agent,speed-test,twingate-connector}/`
     dirs in the repo _exist_ but no `portainer_stack` resource
     references them — they're orphan artifacts.
- **Proposed integration (matches existing pattern):**
  - Convert each to a `portainer_stack` resource scoped to endpoint 14
    (NAS), using `terraform/portainer/stacks/{netbootxyz,speed-test,twingate-nas}.yml.tftpl`.
    Pattern: same as the existing `pihole_3` (NAS endpoint 14) +
    `prometheus_2` resources in `stacks.tf`.
  - For Twingate-NAS: store its `TWINGATE_ACCESS_TOKEN` /
    `TWINGATE_REFRESH_TOKEN` in Vault at
    `secret/homelab/twingate/fresh-marmoset` (matches the existing
    `secret/homelab/twingate/sepia-hornet` and `secret/homelab/twingate/golden-mussel`
    naming) and add a `vault_kv_secret_v2` data source in
    `terraform/portainer/vault.tf`. Wire it in via the stack's `Env`
    map, identical to `twingate_a`/`twingate_b`.
  - Then on NAS, `docker compose down` the legacy stack, fold its
    state in (volumes for netbootxyz config), and Portainer will
    reconcile against TF.
- **Files that would change:**
  - New: `terraform/portainer/stacks/netbootxyz.yml.tftpl`,
    `terraform/portainer/stacks/speed-test.yml`,
    `terraform/portainer/stacks/twingate-nas.yml`
  - Edit: `terraform/portainer/stacks.tf` (3 new resources),
    `terraform/portainer/vault.tf` (1 new data source)
  - New Vault path: `secret/homelab/twingate/fresh-marmoset`

### C3. Bind-mounted live config files NOT in repo

(rproxy Nginx.conf / promtail config / vault config.hcl / registry htpasswd & config.yml)

- **What it is:** Multiple Portainer-TF stacks mount runtime config
  files from `/nfs/dockermaster/docker/...` rather than rendering
  them via Compose `configs:`. The vault stack file (`vault.yml`)
  even has a comment **explicitly documenting** the manual-edit gap
  (lines 6-22).
  - **rproxy** (`reverse-proxy.yml.tftpl`):
    - `/nfs/dockermaster/docker/nginx-rproxy/config/conf/nginx.conf` → `/etc/nginx/nginx.conf`
    - `/nfs/dockermaster/docker/nginx-rproxy/config/promtail-config.yml` → `/etc/promtail/config.yml`
    - `/nfs/dockermaster/docker/nginx-rproxy/bin/start.sh` → `/usr/bin/start.sh` (promtail entrypoint)
  - **vault** (`vault.yml`):
    - `/nfs/dockermaster/docker/vault/vault/config/config.hcl` → `/vault/config/config.hcl`
    - In-place comment says "REQUIRED MANUAL EDIT (NOT IN REPO)".
  - **registry** (`docker-registry.yml`):
    - `/nfs/dockermaster/docker/registry/auth/htpasswd` → `/auth/htpasswd`
    - `/nfs/dockermaster/docker/registry/config.yml` → `/etc/docker/registry/config.yml`
- **Why it's a gap:**
  - "If `/nfs` was wiped right now and I ran `terraform apply` from a
    clean state, would this service come up?" — answer: NO for any
    of these. They'd all start with default upstream config and
    behave wrong (vault would refuse to join the raft cluster
    without `node_id=vault-1` + `retry_join` blocks; registry would
    have no auth; Nginx would serve the upstream default page).
  - These are exactly the bind-mount anti-pattern in the IaC-first
    feedback memory (point #1).
- **Proposed integration (matches existing vhost.d pattern):**
  - **rproxy:** add `configs:` blocks to `reverse-proxy.yml.tftpl`
    (and `-2.yml.tftpl`, `-3.yml.tftpl`) that render
    `nginx.conf` and `promtail-config.yml` via `templatefile()` from
    repo files at `dockermaster/docker/compose/nginx-rproxy/conf/nginx.conf`
    and `.../promtail-config.yml`. The repo dir already exists as a
    historical snapshot; check it's current and use it as source of
    truth. Drop the bind-mounts. The `start.sh` is a 6-line wget+exec
    shim — bake into a custom promtail image or fold into the
    container entrypoint via `command:`.
  - **vault:** render `config.hcl` per-host via `templatefile()` with
    `node_id` and `api_addr` as variables. New file
    `terraform/portainer/stacks/vault-config.hcl.tftpl`, referenced
    via Compose `configs:` in `vault.yml` / `vault-2.yml` / `vault-3.yml`.
    Drop the per-host bind-mount. This also closes the explicit comment
    gap in `vault.yml` lines 6-22 — Phase 3a's "REQUIRED MANUAL EDIT".
  - **registry:** the `htpasswd` file should be sourced from Vault.
    Add `secret/homelab/registry/htpasswd` (single field with the
    file contents — Vault is fine for blob secrets), data source
    in `vault.tf`, and render via `configs.content`. The `config.yml`
    is non-secret — fold into a `configs:` block reading from
    `terraform/portainer/stacks/docker-registry-config.yml.tftpl`.
- **Files that would change:**
  - New: `terraform/portainer/stacks/vault-config.hcl.tftpl`,
    `terraform/portainer/stacks/docker-registry-config.yml.tftpl`,
    promote `dockermaster/docker/compose/nginx-rproxy/conf/nginx.conf`
    - promtail-config.yml as the IaC source (already exist as repo
    snapshots, may need refresh).
  - Edit: `vault.yml`, `vault-2.yml`, `vault-3.yml`,
    `reverse-proxy.yml.tftpl` (+ -2/-3), `docker-registry.yml` to
    use `configs:` instead of bind-mounts. Remove the manual-edit
    comment from `vault.yml`.
  - New Vault path: `secret/homelab/registry` (htpasswd field).

### I4. pfSense firewall rules (118 rules) not in IaC

- **What it is:** 118 firewall rules across 16 interfaces (notably
  opt2 = HOMELAB with 33 rules, opt5 = 13, opt1 = 11, opt3 = 11).
  None are in the repo.
- **Why it's a gap:**
  - The whole point of the pfSense REST API + the recent
    `sync-pfsense-acme-actions.py` "claim what we own" pattern is
    to bring pfSense state into the repo. Firewall rules are by far
    the largest unmanaged surface area.
  - One subtle drift example already memorialized: the
    `feedback_pfsense_api_id_shift.md` memory documents that NAT
    rule IDs shift after PATCH, which made an automated firewall
    sync hazardous — but the same pattern of associating by
    description / `tracker_id` is workable.
  - Tracked in feedback memory as: "⚠️ pfSense HAProxy ACL/use_backend:
    not currently doing service fan-out … the pattern (`scripts/sync-pfsense-acme-actions.py`-style
    YAML+REST reconciler) is reusable when/if HAProxy is repurposed".
- **Proposed integration (incremental):**
  - **Phase 1** (low-risk, high-value): codify the 17 NAT port-forwards
    that were enumerated in this audit (RustServer, HTTPS-to-rproxy,
    DNS redirects, Andre Minecraft) into a new
    `pfsense/nat-port-forwards.yml` source-of-truth, plus
    `scripts/sync-pfsense-nat.py` reconciler. Match by `descr`/`tracker_id`
    not by index, to avoid the id-shift bug.
  - **Phase 2** (later): replicate for firewall rules. Start with
    the WAN interface (only 15 rules, all manual rules — easy to
    enumerate), grow outward.
  - Don't try to bulk-import all 118 rules at once. The "claim
    what we own" pattern from the ACME action_list reconciler
    explicitly leaves anything not in YAML alone — same approach
    here lets us migrate rule-by-rule.
- **Files that would change:**
  - New: `pfsense/nat-port-forwards.yml`,
    `scripts/sync-pfsense-nat.py`.

### I5. `vault-auto-unseal` deployed via `deploy.sh`, not Terraform

- **What it is:** `scripts/vault-auto-unseal/` provides a critical
  systemd oneshot for re-unsealing Vault after host reboots. Deployed
  via `deploy.sh dockermaster` / `dockerserver-1` / `dockerserver-2`,
  which scp's the script + service file + writes
  `/etc/vault/unseal.key` from Keychain.
- **Why it's a gap:**
  - It's a host-side configuration push (systemd unit + binary +
    secret file) — exactly the IaC anti-pattern (point #3 in the
    feedback: "Edit `/etc/foo` on the host then reload"). This was
    written before the IaC-first contract was formalized.
  - If a host is rebuilt or a fourth Docker VM ever joins the
    cluster, deploy.sh has to be remembered and run manually. The
    `2026-04-12 rolling reboot` cited in the README is exactly the
    failure mode this exists to prevent — but the deploy mechanism
    itself has the same "forgot to run it" problem.
- **Proposed integration:**
  - Mirror the `hosts/deploy-network-overrides.sh` pattern: keep
    the script + unit file in `hosts/<hostname>/etc/vault/` (or
    `hosts/_common/etc/vault/` if identical across the 3 hosts)
    and have a top-level `hosts/deploy-system-services.sh` that
    md5-compares + scp's. Same mechanism the network override files
    already use; it's the established pattern even if not strictly
    "Terraform apply".
  - Better: write a small Terraform `null_resource`-free pattern:
    a `local-exec` in `terraform/portainer/` (or new
    `terraform/hosts/`) that depends on the unseal-key Vault data
    source and just calls the existing scp+systemctl. *But:*
    `feedback_no_null_resource.md` explicitly says no `null_resource`
    or local-exec SSH pushes — so the deploy.sh-style approach with
    md5-diff is the one we have and is acceptable until/unless we
    move to a Proxmox Terraform provider that can manage the VM
    itself.
- **Files that would change:**
  - Move `scripts/vault-auto-unseal/{vault-auto-unseal.service,vault-auto-unseal.sh}`
    → `hosts/_common/etc/vault/` (or 3 copies under each host).
  - Extend or sibling-add `hosts/deploy-system-services.sh` (mirror
    of `deploy-network-overrides.sh`). Have it source the unseal
    key from Keychain at apply time.

### N4. Per-host `/etc/netplan/` not synced with `hosts/<host>/etc/systemd/network/`

- **What:** `hosts/README.md` notes that netplan and the systemd
  network override files both define ens19 on each Docker VM, with
  the systemd-network files winning. The netplan file is not in
  the repo.
- **Why:** Low-grade gap. If anyone ever edits the netplan file
  (or a Ubuntu installer rewrites it during an `apt upgrade`),
  drift could emerge. The systemd-network overrides shadow it
  today, but a future kernel/netplan change might not.
- **Fix:** Either commit the netplan files into
  `hosts/<host>/etc/netplan/00-installer-config.yaml` and have
  `deploy-network-overrides.sh` cover them too, or document
  explicitly in the README that netplan is unmanaged-and-shadowed.

### N5. `secret/homelab/paperlessngx` orphaned in Vault

- **What:** Vault path exists but no service consumes it (the
  paperlessngx stack on NAS is dead per finding C2).
- **Why:** Not a critical security issue (it's behind Vault auth)
  but is misleading state.
- **Fix:** Either restore paperlessngx as a TF-managed stack on
  NAS (re-using the secret), or `vault kv delete
  secret/homelab/paperlessngx`.

### N6. ACME deploy actions don't currently push certs to NAS pihole

- **What:** `pfsense/acme-actions.yml` triggers
  `sync-certs-to-rproxy.sh` on renewal of `d.lcamaral.com` and
  `home.lcamaral.com`, which scp's to the 3 dockermaster/ds-1/ds-2
  rproxy nodes. NAS pihole-3 (192.168.4.236) doesn't currently
  receive a cert push (it serves DNS only over plain UDP, so this
  isn't a current problem) — but if/when pihole-3 starts serving
  HTTPS admin UI directly it would.
- **Why:** Not a today-gap; aspirational.
- **Fix:** When pihole-3 needs certs, extend the existing script
  to add NAS as a fourth target. The pattern is already there.
