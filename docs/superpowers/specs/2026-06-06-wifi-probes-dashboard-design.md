# WiFi Probes — Household Coverage Dashboard (Design)

Date: 2026-06-06

## Goal

A single-pane Grafana dashboard visualizing every metric the ESP32-C5 WiFi probes
expose, built so **multiple probes — one per room — can be deployed and compared**
to measure WiFi quality from different areas of the house.

## Multi-room architecture

A probe's room is identified two ways (decision: use **both**):

1. **Prometheus scrape-time `room` label** — _authoritative for the dashboard._ The
   `wifi-probe` scrape job tags each target with `room` + `instance` labels, which
   Prometheus stamps onto **every** metric from that probe. This is the dimension
   all panels template and compare on (`room=~"$room"`, legend `{{room}}`). It is a
   single IaC source of truth, applies uniformly to all 17 metrics, and lets a probe
   move rooms by editing config — no reflash.
2. **Firmware `location` self-ID label** — _complementary._ `CONFIG_WIFI_PROBE_LOCATION`
   (or the captive-portal "Room / location" field → NVS) is emitted as the `location`
   label on `wifi_probe_build_info`. The identity table shows it next to the scrape
   `room` so a mismatch (or unprovisioned blank) is visible — verifying a probe is
   physically where its scrape target claims.

Why not firmware-only: a `location` on `build_info` can't filter the other 16 metrics
without a join, and moving a probe would need a reflash. Why not Prometheus-only: the
device would be anonymous to itself. Both together give authoritative per-room queries
plus self-describing devices.

## Metric inventory (all 17 covered)

| Family | Metrics |
|--------|---------|
| Identity/system | `wifi_probe_build_info{version,idf,chip,location}`, `wifi_probe_uptime_seconds`, `wifi_probe_heap_free_bytes` |
| Link | `wifi_client_connected`, `wifi_client_disconnect_total`, `wifi_client_rssi_dbm{band}`, `wifi_client_channel{band}`, `wifi_client_bssid_info{band,bssid,ssid,auth}` |
| Probe SLA | `probe_success`, `probe_attempts_total`, `probe_success_total`, `probe_duration_seconds_sum`, `probe_duration_seconds_count`, `probe_last_duration_seconds`, `probe_http_status_code`, `probe_last_success_age_seconds` (labels `probe,type,band,target`) |
| Survey | `wifi_ap_rssi_dbm{band,bssid,ssid,channel}` |

## Dashboard structure

`uid: wifi-probes-overview`, datasource `thanos`, schemaVersion 39, `editable: false`.
Five sections in a gap-free 24-column grid:

1. **🚦 Fleet verdict strip** (8 stats) — probes up, rooms covered, worst-room internet,
   weakest RSSI, disconnects/window, min heap, max staleness, probes failing.
2. **🎯 Probe Results SLA** — success matrix (state-timeline), success-ratio + latency
   trends, HTTPS status, a dense per-(room,probe,band) SLA table, freshness bargauge.
3. **📶 Link Quality** — RSSI by room×band, current-RSSI bargauge, current-link table,
   channel-over-time.
4. **🩺 Device Health** (collapsed) — identity/build table, uptime, free-heap leak watch,
   disconnects, connected state-timeline.
5. **📡 AP Survey** (collapsed) — surveyed-AP table, best-AP RSSI per room (bar + trend),
   channel occupancy, airspace inventory.

### Template variables

- `room` (multi) = `label_values(up{job="wifi-probe"}, room)` — primary dimension.
- `band` (multi) = `label_values(probe_success{job="wifi-probe"}, band)` — `2g`/`5g`.
- `window` (interval, default `15m`) — rate/increase window, sized for the band cycle.

### Band-alternating handling

The single radio alternates 5 GHz↔2.4 GHz, so each band's link+probe series refresh
once per ~50–110 s cycle while Prometheus scrapes every 15 s. Therefore:

- counters use `increase(...[$window])` with a generous default (`15m`), never a tight
  fixed window;
- snapshot panels use `last_over_time(...[$window])` so a briefly off-air band keeps its
  last value instead of blanking;
- success/latency ratios use `clamp_min(denominator, 1)` to avoid 0/0;
- the fleet "probes failing" tile uses `... or vector(0)` so a healthy fleet reads a
  green 0 rather than "No data".

## Implementation

Generated by `scripts/grafana/build_wifi_probes_overview.py` (declarative panel specs →
`mk()` renderer → explicit `LINES` layout), mirroring
`build_internet_network_overview.py`. Output:
`terraform/portainer/stacks/grafana-dashboards/wifi-probes-overview.json`.

The panel set came from a multi-agent design pass (one agent per metric family) and was
hardened by a six-lens adversarial review (PromQL, schema, coverage, multi-room, table
joins, IaC).

## Deployment

See [`docs/network/esp32-c5-wifi-probe-dashboard.md`](../../network/esp32-c5-wifi-probe-dashboard.md)
for the per-room rollout. In brief:

1. Flash a probe; set its room via the captive-portal "Room / location" field (or
   `CONFIG_WIFI_PROBE_LOCATION`).
2. Add a target to the `wifi-probe` job in `terraform/portainer/locals.tf` (both replica
   configs) with `labels: { instance: wifi-probe-<room>, room: <room> }`.
3. `terraform apply` (Prometheus + Grafana), then force a Portainer redeploy.

### Constraints / caveats

- Probes must be on a network **reachable from Prometheus** (ds-1 `192.168.48.45`);
  a probe on an isolated SSID won't scrape.
- The `wifi-probe` job ships with **zero targets** (`static_configs: []`), so the `room`
  var is empty and panels read "No data" until the first probe is added — expected.
- Grafana's SQLite-on-NFS corruption issue (tracked separately) can delay the dashboard
  rendering even though the JSON provisions correctly.
